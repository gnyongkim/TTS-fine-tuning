## 3. 최종 추천: **GPT-SoVITS v2 (한국어 파인튜닝)**

### 3.1 추천 사유
1. **요구사항 적합도 1위** - LLM-style(AR-GPT) 구조 + 모바일 친화적 사이즈 + 공식 한국어 지원의 세 조건을 모두 충족하는 거의 유일한 모델.
2. **RTX 4070 Ti(12GB)에 최적** - 파인튜닝 시 약 6~8GB VRAM만 사용. 배치 사이즈 12까지 안정적.
3. **공개 데이터셋과의 궁합** - KSS(약 12시간), AIHub 감정·다화자 데이터셋과 호환되는 전처리 파이프라인이 공식 제공됨.
4. **ONNX 변환 검증** - GPT 부분(`s1`)과 SoVITS 부분(`s2`)을 각각 ONNX로 export하는 공식/커뮤니티 스크립트가 존재(`onnx_export.py`). React Native·Flutter 모바일 데모 사례 다수.
5. **Few-shot 능력** - 본 데이터셋이 부족하더라도 5초~1분 reference로 보조 화자 확장이 가능 → 향후 다화자 확장 시 재학습 부담 ↓.
6. **양자화 후 모바일 사이즈** - INT8 양자화 시 GPT ~80MB, SoVITS ~50MB → 합산 약 130MB로 균형형(100~300MB) 목표 달성.

### 3.2 모델 구성
- **GPT (s1)**: 텍스트 → semantic token (LLM-style autoregressive)
- **SoVITS (s2)**: semantic token + reference embedding → waveform (VITS 계열 디코더)
- **Reference Encoder**: 화자 임베딩 추출

### 3.3 학습/배포 파이프라인 개요
```
[텍스트+오디오 데이터셋]
        ↓
  전처리 (Whisper ASR 정렬, BERT 텍스트 인코딩, HuBERT semantic)
        ↓
  GPT (s1) 파인튜닝   ──┐
  SoVITS (s2) 파인튜닝 ──┴──> 통합 추론 검증
        ↓
  ONNX Export (s1.onnx, s2.onnx, vits.onnx 분리)
        ↓
  ONNX Runtime Mobile + INT8 양자화
        ↓
  Android (NNAPI) / iOS (CoreML EP) 배포
```

## 4. 백업 옵션 (상황별)

### 4.1 더 높은 품질이 필요하고 모바일 사이즈에 여유가 있을 때
- **XTTS-v2 한국어 파인튜닝 + INT8 양자화**
- 자연스러움/감정 표현이 GPT-SoVITS보다 한 수 위
- 단점: 양자화 후에도 600MB+ → 저사양 폰 제외해야 함

### 4.2 다화자/감정 확장을 강하게 원할 때
- **Spark-TTS 또는 Fish-Speech v1.5 한국어 파인튜닝**
- LLM 기반 controllability 우수
- 단점: 한국어 검증 사례 부족, ONNX 변환 안정성 미확인

### 4.3 ONNX 안정성 + 초경량을 원할 때 (LLM-style 포기 가능 시)
- **VITS2 / Piper 한국어 파인튜닝**
- 50MB 이하, ONNX 변환 매우 안정적
- 단점: LLM-style이 아니므로 본 프로젝트 요구사항 미충족
